# Definition of Done (DoD) – Music Player Application

**Version:** 1.0  
**Date:** 2025-11-12  
**Purpose:**  
To define objective criteria for user stories, features, and sprints, so that all deliverables meet the project’s quality, testing, and documentation standards in compliance with **CMMI Level 2**.

---

## 1. Story-Level Completion Criteria

| Area                        | Criterion                                                              | Status |
|-----------------------------|------------------------------------------------------------------------|--------|
| **Requirements**            | Story acceptance criteria implemented exactly as defined.              | ☐ |
|                             | Story linked to GitHub issue with correct ID (e.g., S2-05).            | ☐ |
|                             | `requirements_traceability.csv` updated with code and test links.      | ☐ |
| **Design & Implementation** | Code follows naming conventions and PEP-8 style.                       | ☐ |
|                             | Functions contain docstrings/inline comments.                          | ☐ |
|                             | Code runs on lab machines without external dependencies.               | ☐ |
|                             | All relevant CIs committed and versioned.                              | ☐ |
| **Testing**                 | Unit tests (black-box and white-box) created and pass locally.         | ☐ |
|                             | Symbolic and concolic tests created and executed successfully.         | ☐ |
|                             | Minimum **85 % coverage** achieved (verified in `coverage_report.csv`). | ☐ |
|                             | No critical or major defects open.                                     | ☐ |
| **Quality Assurance**       | Peer review and DoD checklist completed.                               | ☐ |
|                             | Evidence uploaded to `/evidence/MA` and `/evidence/PPQA`.              | ☐ |
| **Documentation**           | Help/CLI documentation updated for new commands in `README.md`.         | ☐ |
|                             | Evidence links included in `requirements_traceability.csv`.            | ☐ |
|                             | Tagged release committed                                               |☐|

---

## 2. Sprint-Level Completion Criteria

| Area | Criterion                                                                           | Status |
|-------|-------------------------------------------------------------------------------------|--------|
| **Planning** | Sprint plan approved and PERT diagram updated.                                      | ☐ |
| **Monitoring** | Burndown, Velocity, and EVM reports completed.                                      | ☐ |
| **Measurement** | Coverage and metrics reports filed in `/evidence/MA/`.                              | ☐ |
| **Configuration Management** | All branches merged, reviewed, tagged (e.g., `v0.x.0`), and logged in `Tag_Log.md`. | ☐ |
| **Quality Assurance** | SCrum Master completed sprint review.                                               | ☐ |
| **CMMI Evidence** | All six process areas (REQM, PP, PMC, CM, MA, PPQA) updated.                        | ☐ |

| Deliverable | Criterion                                                             | Status |
|--------------|-----------------------------------------------------------------------|--------|
